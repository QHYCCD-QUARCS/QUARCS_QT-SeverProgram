## 3. 为什么需要基础化拆分

当前平场法虽然效果很好，但它还带着明显的“导星场景语义”，包括：

- 头文件直接依赖导星类型
- 返回值包含“最佳导星星点”含义
- 候选筛选规则与导星需求耦合
- 选择策略与检测过程混在一起

这会带来几个问题：

1. **调焦无法直接复用**
   当前局部对焦需要的其实是“检测星点 + 按对焦策略选一颗星”，而不是“按导星策略选一颗星”。

2. **不同业务的选星目标不同**
   导星更关注：
   - 高 SNR
   - 不靠边
   - 不近饱和
   - 适合长期跟踪

   局部对焦更关注：
   - 在用户点击点附近
   - 在 ROI 中心附近
   - HFR/HFD 合适
   - 不过曝
   - 背景稳定

3. **调试成本高**
   如果检测和选星混在一起，后续排查时很难分辨：
   - 是没检测到星
   - 还是检测到了，但策略没有选它

4. **后续难以统一**
   如果不先拆底层，后续每个业务都容易再包一层自己的“变种平场法”。

因此建议把当前平场法提炼为：

- 一套**基础检测能力**
- 一套或多套**业务选星策略**

## 4. 目标拆分原则

建议按下面三层拆分，而不是只移动文件位置。

### 4.1 第一层：基础检测层

职责：

- 输入图像
- 生成 self-flat-field
- 扣背景
- 找峰值
- 去重
- 质心精化
- 计算每颗候选星的基础度量

这一层不负责：

- 决定“哪颗最适合导星”
- 决定“哪颗最适合局部对焦”
- 管理锁星状态
- 管理跨帧跟踪

### 4.2 第二层：业务选星层

职责：

- 从第一层输出的候选星点中
- 按场景策略选出一颗

例如：

- 导星策略
- 局部对焦策略
- 用户点击附近优先策略
- ROI 中心优先策略

### 4.3 第三层：跟踪/状态层

职责：

- 锁定上一帧目标
- 跨帧追踪
- 丢星恢复
- ROI 自动居中

这层应该继续留在：

- `GuiderCore`
- `MainWindow::selectStar(...)`

而不应该混进基础平场法工具中。

## 5. 输出建议拆成两级

建议采用两级输出模型。

### 5.1 第一级：输出所有候选星点

基础检测层输出一个 `DetectionResult`，包含：

- 所有原始候选
- 去重后候选
- 通过基础规则筛选后的候选
- 被拒绝的候选及原因

这一级回答的问题是：

> 图里有哪些候选星点，它们各自的质量如何？

### 5.2 第二级：从候选星点中选出一颗

业务选星层再输出一个 `SelectionResult`，包含：

- 是否选中
- 选中了哪颗
- 为什么选它
- 评分是多少
- 候选排序结果

这一级回答的问题是：

> 在这些候选星里，这次为什么选中这一颗？

这种拆法的优点是：

- 导星可复用同一检测底座
- 调焦可复用同一检测底座
- 脚本模式可先看候选，再决定选星策略
- 调试时可以精确区分“检测失败”和“策略未命中”

## 6. 输入设计建议

除了图像本身，建议输入分成四组。

### 6.1 图像上下文 `ImageContext`

建议字段：

- `imageWidth`
- `imageHeight`
- `bitDepth`
- `channels`
- `binningX`
- `binningY`
- `roiOriginX`
- `roiOriginY`
- `pixelScaleArcsecPerPixel`
- `isColor`

用途：

- 对导星来说，`pixelScaleArcsecPerPixel` 有助于后续角秒尺度调参
- 对局部对焦来说，`roiOriginX / roiOriginY` 用于把 ROI 内坐标映射回全图
- `binning` 有助于统一尺度解释

### 6.2 检测参数 `DetectParams`

建议字段：

- `kernelSize`
- `flatMethod`
- `gaussianSigma`
- `snrThreshold`
- `minHfd`
- `maxHfd`
- `minHfr`
- `maxHfr`
- `minSeparationPx`
- `edgeMarginPx`
- `nearSaturationRatio`
- `centroidHalfSize`
- `centroidKSigma`
- `maxCandidates`
- `computeHfr`

用途：

- 平场法内部参数
- 候选过滤基本阈值
- 是否在基础层顺带计算 `HFR`

### 6.3 搜索区域 `SearchRegion`

建议字段：

- `includeRect`
- `excludeRects`

后续如有需要可扩展：

- `binaryMask`

用途：

- 导星可全图搜索
- 局部对焦可只在 ROI 内搜索
- 用户点击后可只在点击点附近局部重检

### 6.4 选星策略 `SelectionPolicy`

建议字段：

- `mode`
- `minSnr`
- `maxHfd`
- `maxHfr`
- `avoidSaturated`
- `minEdgeDistPx`
- `preferredX`
- `preferredY`
- `hasPreferredPoint`
- `preferCentralStar`
- `centerWeight`
- `snrWeight`
- `hfrWeight`
- `peakWeight`

用途：

- 区分“检测”和“业务决策”

## 7. 输出设计建议

### 7.1 基础星点结构 `DetectedStar`

建议最少字段：

- `x`
- `y`
- `fullX`
- `fullY`
- `snr`
- `hfd`
- `hfr`
- `peakADU`
- `flux`
- `bgMean`
- `bgStd`
- `edgeDistPx`
- `saturated`
- `centroidRefined`

说明：

- `x/y` 为当前输入图坐标
- `fullX/fullY` 为还原后的全图或传感器坐标
- `hfd` 更贴近现有导星平场法
- `hfr` 更贴近后续局部对焦需求
- `bgMean/bgStd` 对调试和后续评分都很有帮助

### 7.2 被拒绝候选 `RejectedStar`

建议字段：

- `DetectedStar star`
- `RejectReason reason`
- `std::string detail`

`RejectReason` 建议至少支持：

- `LowSnr`
- `BadHfd`
- `BadHfr`
- `NearEdge`
- `NearSaturation`
- `CentroidFailed`
- `Duplicate`
- `OutsideSearchRegion`
- `InvalidShape`

### 7.3 检测结果 `DetectionResult`

建议字段：

- `allCandidates`
- `dedupCandidates`
- `validCandidates`
- `rejectedCandidates`
- `debug`

其中 `debug` 可至少包含：

- `rawPeakCount`
- `dedupCount`
- `validCount`
- `rejectedCount`
- `summary`

### 7.4 选星结果 `SelectionResult`

建议字段：

- `success`
- `selected`
- `rankedCandidates`
- `summary`

其中 `selected` 可进一步包含：

- `DetectedStar star`
- `sourceIndex`
- `score`
- `reason`

## 11. 第一阶段实施边界建议

为了降低风险，第一阶段不建议同时做以下事情：

- 不要立刻改 `AutoFocus` 主状态机
- 不要立刻替换 Python HFR/SNR 流程
- 不要立刻改导星跨帧跟踪逻辑

第一阶段只建议做到：

1. 基础平场法检测独立成模块
2. 导星用新模块但保持现有行为
3. 提供可被局部对焦试验调用的候选星列表输出

## 12. 第二阶段与局部对焦的关系

在后续“局部对焦”实施中，这个基础模块最适合承担的职责是：

1. 在 ROI 或全图中找出候选星点
2. 返回所有候选及其质量指标
3. 根据用户点击点或 ROI 中心，选出一颗作为对焦目标

而这些内容不应由基础模块承担：

1. 维护跨帧锁星状态
2. 决定何时更新 ROI
3. 决定何时启动/停止对焦流程

这些仍应由：

- `MainWindow`
- `AutoFocus`
- 局部对焦控制层

负责。

## 13. 最小实施建议

如果本次只做一轮重构，推荐的“最小完成目标”是：

1. 新建 `src/star_detect/`
2. 抽出通用类型与参数结构
3. 把平场法实现迁成纯检测接口
4. 导星继续复用这套接口
5. 新增一个简单的“按策略选一颗星”的接口

做到这一步后：

- 不会破坏当前导星逻辑

## 14. 结论

当前平场法效果已经具备作为基础能力沉淀的价值。

但为了真正服务后续：

- 导星
- 局部对焦
- 自动局部对焦

建议不要仅做“文件移动”式重构，而应按：

```text
基础检测层 -> 业务选星层 -> 状态跟踪层
```

进行拆分。

其中最关键的设计点是：

- **输入除了图像，还要显式携带图像上下文、检测参数、搜索区域与选星策略**
- **输出拆成两级：先输出所有候选星点，再从中选出一颗**

这是后续稳定演进、复用与调试成本最低的方案。
