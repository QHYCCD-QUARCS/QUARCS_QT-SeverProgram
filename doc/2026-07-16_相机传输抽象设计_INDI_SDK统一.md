# 相机传输抽象设计：INDI / SDK 行为级统一（ICamera）

状态：设计草案（待评审后分期实施）。作者协作：QUARCS 重构。

## 1. 问题与目标

当前每个相机**行为**都以 `if (isSDKConnect) { SDK 实现 } else { INDI 实现 }` 的成对分支
散落在 ~560 处（`mainwindow_device_connection.cpp` 就 277 处，涉及 17 个文件）。
后果：

- **模式切换（INDI↔SDK）易留脏状态**：切换要“拆掉 INDI 整套状态 + 搭起 SDK 整套状态”，
  命令式散落完成，漏一片就两套状态并存（INDI blob 回调仍触发 + SDK 定时器也轮询，
  或 dp 未清导致某行为仍走 INDI）。这是“两相机都连 INDI、断开一个再切 SDK 出乱”的根因。
- 每角色状态分裂成两套并行：INDI（`dp*` + slot 标志）与 SDK（全局
  `sdkXxxHandle`/`g_sdkXxxPoolIndex`/`SdkSerialExecutor`/定时器/ready 标志），靠
  `isSDKConnect` 选活的，手工保持同步。

**目标**：把“分派 SDK/INDI”从 ~560 处收敛到**一次**——每角色持有一个 `ICamera` 策略对象，
所有行为通过接口调用；建对象时决定用哪套实现。等价性由结构强制（对齐既有原则
[[project-quarcs-auto-decision-equivalence]]：两条必须等价的路径，在单一接缝处分派）。

## 2. 等价性判定（依据代码审计）

- **操作语义层 = 等价**：两条路对外契约相同，近 1:1 对应：

  | 行为 | SDK 命令 | INDI 方法 |
  |---|---|---|
  | 芯片信息 | GetChipInfo | getCCDBasicInfo |
  | 增益 | SetGain/GetGain | setCCDGain/getCCDGain |
  | 偏置 | SetOffset/GetOffset | setCCDOffset/getCCDOffset |
  | 分辨率/ROI | SetResolution/GetEffectiveArea | setCCDFrameInfo/setCCDBasicInfo |
  | 曝光 | StartSingleExposure/SetExposure | takeExposure |
  | 取帧 | GetSingleFrame（轮询） | blob 回调（异步） |
  | 中止 | CancelExposure/Abort | setCCDAbortExposure |
  | 温控 | SetCoolerTargetTemperature/GetTemperature | setTemperature/getTemperature |
  | USB 流量 | SetUsbTraffic/GetUsbTraffic | setCCDUsbTraffic/getCCDUsbTraffic |
  | Bin | SetBinMode | setCCDBinnign |
  | 彩色/CFA | IsColorCamera | getCCDCFA |
  | Live/流 | BeginLive/StopLive/SetStreamMode | 连续曝光/stream |

- **实现机制层 ≠ 等价**：INDI 异步回调 vs SDK 工作线程轮询；线程模型不同；拆卸流程不同。
  → 抽象边界取在**操作语义层**；机制差异封进各实现内部。唯一需特殊处理的是**取帧**。

## 3. 接口设计

```cpp
// icamera.h —— 相机行为接口（语义层），屏蔽 INDI/SDK 机制差异
struct ChipInfo { int maxX=0, maxY=0, bitDepth=0; double pixelSize=0, pixelX=0, pixelY=0; };
struct FrameData { /* 像素缓冲 + 元数据（尺寸/位深/时间戳/frameId） */ };

class ICamera {
public:
    virtual ~ICamera() = default;              // 析构即彻底拆卸（RAII，见 §5）

    virtual bool connect() = 0;
    virtual void disconnect() = 0;
    virtual bool isConnected() const = 0;

    virtual ChipInfo chipInfo() = 0;
    virtual bool isColor() = 0;                 // CFA/彩色
    virtual void setRoiFull() = 0;
    virtual void setRoi(int x,int y,int w,int h) = 0;
    virtual void setBin(int bx,int by) = 0;
    virtual void setGain(int) = 0;   virtual int gain() = 0;
    virtual void setOffset(int) = 0; virtual int offset() = 0;
    virtual void setUsbTraffic(int) = 0;
    virtual void setTargetTemperature(double) = 0; virtual double temperature() = 0;

    virtual void startExposure(double seconds) = 0;   // 单帧
    virtual void abort() = 0;
    virtual void beginLive() = 0;  virtual void stopLive() = 0;

    // 取帧统一出口：无论回调还是轮询，帧就绪时回调此函数。
    // INDI 实现把 blob handler 接到这里；SDK 实现把定时器轮询结果接到这里。
    std::function<void(const FrameData&)> onFrame;
};

class IndiCamera : public ICamera { /* 持 indi_Client + INDI::BaseDevice* dp */ };
class SdkCamera  : public ICamera { /* 持 SdkManager 句柄 + poolIndex + SdkSerialExecutor + timer */ };
```

- 每角色状态（dp / handle / poolIndex / timer / ready 标志 / camId）→ 变成对应实现的**成员**，
  不再是分裂的全局，无需手工同步。
- 角色注册表持有 `std::unique_ptr<ICamera> cameras[MainCamera|Guider|PoleCamera]`。

## 4. 迁移分期（每步交叉编译 + 真机验，独立提交）

- **P0 骨架（零行为变化）**：加 `icamera.h` + `IndiCamera`/`SdkCamera`，方法体先原样搬运既有
  `indi_Client->…` / SDK `call(…)` 逻辑；连接时按 `isSDKConnect` 建对象挂到角色注册表；
  **暂不替换任何调用点**（对象与旧代码并存）。
- **P1 迁同步行为**：把 chipInfo/gain/offset/roi/bin/temp/usbTraffic/abort 等**同步**行为的
  ~N 处 `if(isSDK){}else{}` 逐个换成 `cam->op()`。易验、低风险。
- **P2 迁曝光 + 取帧（最难）**：把取帧统一到 `onFrame`；IndiCamera 接 blob→onFrame，
  SdkCamera 接定时器轮询→onFrame。迁曝光/取图主链路。单独、谨慎、重点真机验。
- **P3 连接/断开/切换**：connect=建实现；disconnect=析构实现（RAII）；模式切换=换实现指针。
  删掉散落的状态迁移代码。**切换 bug 在此消失**。
- **P4 清扫**：`isSDKConnect` 退化为“建哪种实现”的一次性判断；移除 ~560 处分支。

## 5. 为什么这样能根治切换 bug

模式切换 = `slot.camera.reset(new SdkCamera(...))`。旧对象析构时，RAII 确定性拆掉**它自己**的
资源（停定时器、abort 曝光、关句柄 / 解 blob 回调）。不存在“漏拆某片”→ 不可能两套状态并存。
新对象干净初始化。切换正确性由类型系统与析构语义保证，而非人工记账。

## 6. 风险与边界

- 大改（~560 点 + 两套线程模型），必须**增量**，每步真机验；不可一次性重写。
- 最难点在 P2 取帧统一（回调 vs 轮询的线程/生命周期）。建议 P2 前先把 P0/P1 稳住。
- 本设计只覆盖**相机**三角色；Focuser/CFW/Mount 的 SDK/INDI 分派可后续按同法处理。
- 与分配重构解耦：`planCameraAllocation` 决策产出 (角色,设备)，落地时创建对应 `ICamera`。


## 7. 传输混用与扫描约束（2026-07-16 决定，QHY 专家确认）

**结论：同驱动 QHY 相机允许"一台 SDK、一台 INDI"（不同物理相机）。** 现有
"同驱动组全 SDK 或全 INDI"的锁（mainwindow_commands_device.cpp SetConnectionMode
sameDriverCameraGroup 分支）**过宽，可放开**；约束收窄为"同一物理相机不得被双开"
（分配逻辑已保证）。

依据（USB/库层）：
- 同一 libqhyccd.so 两进程加载，库全局状态每进程私有（数据段 COW），不耦合。
- 排他冲突只在 OpenQHYCCD/interface claim，按物理设备；同一台相机双开才冲突。
- 唯一不同相机间耦合点是 ScanQHYCCD：枚举所有相机 → 每台一个 EP0 请求需应答。
  低频无害；高频扫描 + 相机侧高频 EP0 才可能缓存满→死锁。扫描一次不会触发。

**两条硬约束（重构必须遵守）：**
1. **扫描：每次都扫，不加缓存（2026-07-16 用户最终决定）。** EP0 耦合在实际频次下
   良性，而缓存的"何时失效"是难题、易引入 staleness bug，不值得。已把 4 处重复
   ScanCameras 收敛到单点 sdkScanQhyCameras（commit ba5b524，纯 dedup、无缓存）。
   **不再加扫描缓存。**
2. **SDK 只 open 被分配相机**：废除"扫到几台 open 全部"（ConnectAllDeviceOnce ~226），
   只 open 分配给 SDK 角色的相机，其余留给 INDI，避免自造双开冲突。

**落地顺序（在 P0 骨架之上）：**
- 先做 scan 收敛（一次/缓存/共享）+ open-only-assigned —— 混用的前置，且独立降 EP0 风险；
- 再放开锁（保留同一相机不双开）；
- 配合 ICamera 每角色独立传输，去掉"同驱动联动"。
- "INDI→SDK 列表 4→8 重复"（SetConnectionMode 缺对称清理）在此一并解决。
