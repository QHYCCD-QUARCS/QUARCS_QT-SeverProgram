## 7. 对升级工作的判断

### 7.1 当前判断

从现有信息看，项目并不是“完全不能升级”，而是：

- 旧兼容接口仍然能编
- 但 warning 已经大量暴露出 API 风格老化
- 越往后拖，未来迁移成本越可能继续上升

### 7.2 当前最现实的升级切入点

建议先不要一上来全仓升级全部 INDI 用法，而是按下面顺序推进：

1. 先改 `myclient.cpp`
2. 再改 `mainwindow_device_connection.cpp`
3. 再看 `mainwindow_focuser.cpp` 等次级文件

原因：

- `myclient.cpp` 是项目内 INDI 包装最集中的地方
- 先在这里形成一套“现代属性 API 写法样板”，后续复制到其他文件会更顺

### 7.3 关于“开发机编译”与“业务机构建”的判断

当前更推荐把 INDI 升级理解为“两地并行、职责分离”：

- 开发机 `x86`：
  - 更新独立 `indi` / `indi-3rdparty` 仓库
  - 用于阅读上游改动、接口迁移、无硬件条件下的编译验证
- 业务机 `ARM`：
  - 使用独立新目录保存最新源码
  - 在树莓派本机完成真实依赖、真实 SDK、真实硬件链路的构建与验证

这样做的原因是：

- `INDI` 与 `indi-3rdparty` 含有大量硬件驱动和 SDK，天然与处理器架构、系统库、动态库路径有关
- 业务机当前环境还存在 `1.9.9` 包残留与 `2.0.8` 源码安装叠加的历史状态
- 对这类依赖栈来说，树莓派本机源码构建比“开发机编好后搬过去”更稳妥

结论：

- 开发机负责“升级开发与预验证”
- 业务机负责“最终构建、硬件联调与验收”

这样后续若开发机仍误命中 `opencv249`，会在 CMake 配置阶段直接给出明确错误，而不会在源码中后期爆出一批误导性的编译错误。

#### 业务机（ARM）验证结论

业务机已使用新 `INDI` staging 进行 Qt 服务端原生构建验证：

```bash
cmake .. \
  -DQUARCS_INDI_PREFIX=/home/quarcs/Projects-2.2.x/stage-indi/usr/local \
  -DCMAKE_BUILD_TYPE=Release

cmake --build . -j4 --target client
```

结果：

- 命中的 OpenCV 为 `3.4.14`
- `client` 已在业务机构建通过：
  - `[100%] Built target client`

当前判断：

- Qt 服务端已经能够在业务机上对接新 `INDI 2.1.9` 头库并完成原生编译
- 到本轮为止，未发现新的“由 INDI 2.1.9 API 变化直接导致的 Qt 编译失败”
- 当前开发机侧的主要阻塞点不是 INDI，而是本地 OpenCV 版本过旧

### 8.3 第三阶段：视情况引入更新版 INDI

在现有代码先基本脱离旧兼容层后，再决定是否：

- 升 sysroot 中的 INDI
- 升树莓派运行时的 INDI
- 升宿主机本地的 INDI 开发环境

这样风险更可控。
