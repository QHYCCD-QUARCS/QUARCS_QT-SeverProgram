# 设备绑定逻辑总结与修改指南（MainCamera / Guider / INDI / SDK）

本文总结 `QUARCS_QT-SeverProgram` 当前设备绑定行为，尤其是相机（MainCamera/Guider）在 INDI 与 SDK 下的自动分配与手动分配逻辑，作为后续迭代的基线说明。

## 1. 总体原则

当前实现遵循以下优先级：

1. 历史回绑优先（按 `systemdevicelist.system_devices[*].DeviceIndiName` 匹配）。
2. 历史回绑失败时，进入 QHY 规则兜底（仅 QHY 双相机场景）。
3. 仍无法确定时，才进入待分配列表并弹出绑定窗口。

额外约束：

1. 单设备连接请求（`ConnectDriver`）只处理请求角色，不应顺带绑定其它角色。
2. 单角色请求下，不应因同驱动“其它 CCD”自动触发绑定窗口。

---

## 2. 入口与职责

### 2.1 全连接入口（一次性连接）

- 函数：`MainWindow::ConnectAllDeviceOnce(...)`
- 相关续处理：`continueConnectAllDeviceOnce(...)`
- 关键位置：`src/mainwindow.cpp`（约 10k~12k 行）

职责：

1. 扫描并连接 SDK 设备池（相机/电调）。
2. 完成 INDI 设备连接后，进行多设备自动回绑与待分配上报。
3. 适用于“全量连接”场景，允许同时处理多角色。

### 2.2 单设备连接入口

- 函数：`MainWindow::ConnectDriver(MyClient *client, QString DriverName, QString DriverType)`
- 关键位置：`src/mainwindow.cpp`（约 24k~26k 行）

职责：

1. 只连接用户当前请求的一个设备类型（如 MainCamera 或 Guider）。
2. 在同驱动多设备下优先自动选择当前角色目标；必要时才进入分配窗口。

### 2.3 手动绑定入口

- 函数：`MainWindow::BindingDevice(QString DeviceType, int DeviceIndex)`
- 关键位置：`src/mainwindow.cpp`（约 11.8k 行）

职责：

1. 响应前端设备分配面板操作。
2. 支持 SDK 池索引与 INDI 索引绑定。
3. 支持 Main/Guider 交换绑定。

---

## 3. QHY 双相机兜底规则（当前统一规则）

当满足以下条件才触发：

1. 历史回绑未完成目标角色绑定；
2. 处于 QHY 相机双角色场景（MainCamera+Guider）；
3. 可用相机数量 >= 2。

规则：

1. 若存在 `5III` 与非 `5III` 混合：
- `5III -> Guider`
- 非 `5III -> MainCamera`

2. 若两台同系列（都 `5III` 或都非 `5III`）：
- 分辨率高 -> `MainCamera`
- 分辨率低 -> `Guider`

分辨率来源：

1. SDK：`GetChipInfo.maxImageSizeX * maxImageSizeY`
2. INDI：`getCCDBasicInfo(maxX, maxY)` 的 `maxX * maxY`

---

## 4. INDI 绑定链路现状

### 4.1 全连接（continueConnectAllDeviceOnce）

1. 先按历史设备名自动回绑 MainCamera/Guider/PoleCamera。
2. 若 Main/Guider 缺失且满足 QHY 条件，执行 QHY 兜底自动绑定。
3. 未绑定设备继续上报 `DeviceToBeAllocated:*`。
4. 若存在待分配项，发送 `ShowDeviceAllocationWindow`。

### 4.2 单连接（ConnectDriver）

1. 仅处理当前请求角色（MainCamera/Guider/PoleCamera）。
2. 历史回绑失败时，仅对该请求角色应用 QHY 兜底。
3. 已修复：单角色请求时，跳过“其它 CCD”待分配上报，避免误弹绑定窗口。

---

## 5. SDK 绑定链路现状

### 5.1 全连接（ConnectAllDeviceOnce）

1. 扫描并打开相机池。
2. 历史回绑优先。
3. 历史未命中时使用 QHY 规则选 Main/Guider。
4. 仍有未分配才弹窗口。

### 5.2 单连接（ConnectDriver: MainCamera/Guider:SDK）

1. 扫描并打开候选相机。
2. 历史回绑优先。
3. 历史未命中时：
- MainCamera 分支：优先非 5III；同系列取高分辨率。
- Guider 分支：优先 5III；同系列取低分辨率。
4. 选不中才 `ConnectDriverPendingAllocation:*` + `ShowDeviceAllocationWindow`。

---

## 6. 关键消息（前后端协议）

1. `DeviceToBeAllocated:<Type>:<Index>:<Name>`：上报候选设备。
2. `ShowDeviceAllocationWindow`：前端弹设备分配面板。
3. `ConnectDriverPendingAllocation:<Role>`：当前连接进入待分配状态。
4. `ConnectDriverSuccess:<DriverName>`：连接并完成绑定。
5. `BindDeviceTypeList` / `BindDeviceList`：前端刷新绑定状态列表。

---

## 7. 修改建议（后续扩展）

1. 新增规则时，务必同时评估四条路径：
- SDK 全连接
- SDK 单连接
- INDI 全连接
- INDI 单连接

2. 明确“角色边界”
- 单连接请求禁止跨角色副作用（不要顺带绑其它角色）。

3. 保持“历史优先”不被覆盖
- 兜底规则只填缺失角色，不覆盖历史成功回绑。

4. 对外可观测性
- 每个自动绑定决策必须有明确日志（命中规则、候选、最终选择）。

5. 回归验证建议
- 双相机：`QHY600 + QHY5III`
- 双相机同系列：`QHY5III + QHY5III`（或非 5III + 非 5III）
- 分别验证：
  1. MainCamera 单连接
  2. Guider 单连接
  3. 全连接
  4. 手动分配交换

---

## 8. 当前已知风险点

1. 日志中 `Each Device Only Has One:0` 并不等价于一定会弹窗，需要结合 `hasPendingAllocation` 判断。
2. INDI 驱动属性读取偶发 `CCD_INFO/CCD_CFA` 不完整时，会影响分辨率判定质量；当前已做兜底排序（无分辨率记为较低优先级）。
3. SDK 相机池与角色句柄共享，重连时要避免误清理其它角色使用中的句柄（当前通过 pool-in-use 分支保护）。
