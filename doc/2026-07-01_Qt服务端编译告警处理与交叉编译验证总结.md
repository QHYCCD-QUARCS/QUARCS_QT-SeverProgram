# Qt 服务端编译告警处理与交叉编译验证总结

最后更新：2026-07-01

## 1. 背景

本次工作的直接目标是：

- 在 `QUARCS_QT-SeverProgram` 仓库中实际执行一次 Qt 服务端编译
- 收集当前真实编译输出中的 warning / error
- 优先处理会阻塞编译、会明显放大噪声、或已经暴露接口老化迹象的问题

本轮过程中同时验证了两条构建路径：

- 本机原生构建：`src/build-native`
- 树莓派交叉构建：`build-rpi-verify`

最终确认：

- 本机原生构建不适合作为当前 warning 收敛基线
- 交叉构建更接近真实交付环境，应作为后续 warning / INDI 升级工作的主验证入口

## 2. 本轮关键结论

### 2.1 交叉构建 `client` 已重新打通

本轮最终在以下目录完成了 `client` 目标构建通过：

- `build-rpi-verify`

关键命令：

```bash
cd /home/q/workspace_origin/QUARCS_QT-SeverProgram/build-rpi-verify
cmake -S ../src -B .
make -j4 client
```

最终结果：

- `[100%] Built target client`

说明：

- 当前 Qt 服务端主目标 `client` 在交叉编译链路上已可再次完成构建
- 后续 warning 清理、INDI 版本升级验证，建议继续以该目录和该命令作为主入口

### 2.2 本机原生构建暴露了本机环境与项目依赖的历史兼容问题

本机原生构建目录：

- `src/build-native`

本轮排查中发现：

- 本机 OpenCV 更接近旧版 2.4 体系
- 代码已大量使用较新的 OpenCV API 与头文件布局
- 因此 native 构建会暴露大量“本机环境版本过旧”带来的问题

这些问题包括但不限于：

- `opencv2/imgcodecs.hpp` 不存在
- `cv::LINE_AA` / `cv::LINE_8` 差异
- `connectedComponentsWithStats`
- `estimateAffine2D`
- `estimateAffinePartial2D`

结论：

- `src/build-native` 更适合做本机兼容性排查
- 不适合继续作为当前 warning 主基线

## 3. 本轮已处理的问题

### 3.1 CMake / sysroot / 三方依赖路径问题

为了让交叉构建回到可用状态，本轮对 `src/CMakeLists.txt` 做了若干补强：

- 补了 `QHYCCD` 头文件路径自动探测
- 补了 `StellarSolver` 头文件路径自动探测
- 把 `OpenCV` / `StellarSolver` / `QHYCCD` 相关 include 更明确地下发到目标
- 对 `StellarSolver::stellarsolver` 的 imported target 做了 sysroot 路径修正

其中最关键的一点是：

- `StellarSolverConfig.cmake` 内部给出的依赖库路径是 `/usr/lib/...`
- 交叉编译时如果不映射到 `${CMAKE_SYSROOT}/usr/lib/...`，会在链接阶段出现找不到库的问题

本轮已处理掉的典型报错：

- `No rule to make target '/usr/lib/aarch64-linux-gnu/libcfitsio.so', needed by 'client'`

### 3.2 Qt / OpenCV / 旧接口兼容问题

为了让本机和交叉构建都能更稳定地进入后续阶段，本轮做了以下兼容修补：

- `tools.h`
  - 对 `opencv2/imgcodecs.hpp` 做了 `__has_include` 兼容处理
  - 增加了旧版 / 新版 OpenCV 线型常量兼容封装
- `tests/guiding_batch_analyzer.cpp`
  - 同步处理 `imgcodecs.hpp` 与线型常量兼容
- `mainwindow.cpp`
  - 把 `cv::LINE_8` 改成兼容封装
- `tools.cpp`
  - 把 `QMatrix` 相关调用改为 `QTransform`
  - 把旧式 `QProcess::start(QString)` 改为更稳的参数形式
- `websocketclient.cpp` / `websocketclient.h`
  - 去掉了已弃用的 `QNetworkConfigurationManager`

### 3.3 明确 warning 修补

本轮已直接修掉的 warning 类型主要有：

- 非 `void` 函数缺缺省返回值
- `NULL` 赋给非指针对象
- `char *` 接收字符串字面量
- 属性有效性判断使用旧式 `!= nullptr`

已处理的代表性文件：

- `src/mainwindow_mount_control.cpp`
- `src/mainwindow_device_connection.cpp`
- `src/autopolaralignment.cpp`
- `src/myclient.cpp`

### 3.4 交叉链接 ABI 问题

本轮还遇到一个很关键的交叉链接问题：

- `std::stoi` 在当前交叉链路下引入了 `__isoc23_strtol`
- 树莓派 sysroot libc 中不提供该符号

实际报错：

- `undefined reference to '__isoc23_strtol'`

已处理方式：

- 把 `src/myclient.cpp` 中 `getCCDCFA()` 的 `std::stoi` 改为 `QString::toInt()`
- 同时增加了转换失败保护日志

这一步是本轮让 `client` 最终完成链接的关键修复点之一

## 4. 当前剩余 warning 的主要结构

在交叉构建重新打通后，当前剩余 warning 已明显收敛，主要分为两类。

### 4.1 与项目代码直接相关，但风险较低的 warning

主要是 GCC 12 关于 C++17 ABI 的提示：

- `parameter passing for argument of type 'std::pair<...>' when C++17 is enabled changed to match C++14 in GCC 10.1`

这类输出当前表现为 `note`，主要集中在：

- `guiding/GuidingStarDetector.cpp`
- `guiding/flatfield/FlatFieldDetector.cpp`
- `focused_star_detection.cpp`
- `autopolaralignment.cpp`

这类问题当前不是功能阻塞项，可后置处理。

### 4.2 INDI 旧兼容接口带来的弃用 warning

这是当前 warning 的主噪声来源。

特点：

- 代码仍大量使用 `INDI_PROPERTY_BACKWARD_COMPATIBILE` 保留下来的旧式访问方式
- 比如：
  - `property->np[i].value`
  - `property->tp[i].text`
  - `property->getState()`
  - `property->getStateAsString()`
  - `INDI::Property` / `INDI::PropertyNumber` 的指针式访问

本轮为了先恢复稳定构建，对部分文件做了局部 `-Wno-deprecated-declarations` 降噪：

- `myclient.cpp`
- `mainwindow_device_connection.cpp`
- `mainwindow_mount_control.cpp`
- `autopolaralignment.cpp`

说明：

- 这是临时降噪措施，不是最终升级方案
- 后续如果继续推进 INDI 升级，仍需要把旧式访问逐步改为现代 API 写法

## 5. 与 INDI 升级工作的关系

本轮编译 warning 处理，实际已经暴露出一个很明确的方向：

- 当前 INDI 使用方式明显偏旧
- 当前代码可以在兼容层上继续编，但 warning 已经足够多
- 如果后续直接推进到更新的 INDI 版本，`myclient.cpp` 会是第一优先级改造文件

当前相关文档请继续看：

- `doc/2026-07-01_INDI版本现状对比与升级工作入口.md`

## 6. 后续建议

建议后续继续按下面顺序推进：

1. 固定以 `build-rpi-verify` 的 `make -j4 client` 作为主验证入口
2. 把 `myclient.cpp` 作为 INDI 现代 API 改造的第一站
3. 再处理 `mainwindow_device_connection.cpp`
4. 最后再决定是否撤掉局部 `-Wno-deprecated-declarations`

不建议当前优先做的事：

- 继续投入大量时间在 `src/build-native` 上追本机 OpenCV 2.4 兼容
- 在还未完成 INDI 接口迁移前，把所有警告都强行压到 0
