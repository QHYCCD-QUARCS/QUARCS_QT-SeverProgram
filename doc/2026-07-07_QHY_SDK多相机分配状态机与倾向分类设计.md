# 2026-07-07 QHY SDK多相机分配状态机与倾向分类设计

## 1. 背景

当前 QUARCS 在 `QHYCCD + SDK` 多相机场景下，存在以下典型问题：

- 主相机、导星、极轴镜各自存在独立的扫描/自动回绑/自动选相机逻辑
- 多相机时，即使已经自动绑定成功，后端仍会继续弹出设备分配窗口
- 手工绑定与自动绑定的边界不清晰，后续连接流程可能覆盖用户选择
- 导星链路会再次按“导星倾向”自动挑选相机，导致前面已选好的相机被改写
- `DEMO` 相机当前缺乏明确产品规则，容易在实现中被当成“特殊测试设备”单独处理

这些问题叠加后，会表现为：

- 设备连接反复触发
- 多相机分配窗口自动乱跳
- 用户明明已经选了 `DEMO`，实际导星曝光却落到了 `QHY5III585`

本文目标是给出一套完整、统一、可实施的 `QHYCCD + SDK` 多相机连接与分配设计，作为后续代码重构依据。

## 2. 设计目标

本次设计仅讨论“相机”。

不讨论：

- 赤道仪
- 电调
- 滤镜轮
- 历史配置兼容细节的最终实现

本次重点目标：

1. 从“左侧设备栏目全空，什么都没选”的最干净状态出发，建立完整逻辑
2. 对 `QHYCCD + SDK` 建立唯一的扫描、建议、分配、锁定流程
3. 将所有 QHY 相机视为标准相机，不再把 `DEMO` 作为特殊设备分支处理
4. 引入“相机倾向分类”作为自动建议依据
5. 明确 `PoleMaster` 的唯一性规则：一旦存在，直接归属于极轴镜角色，不参与人工选择

## 3. 核心原则

### 3.1 唯一扫描原则

`QHYCCD + SDK` 多相机模式下，只允许存在一套统一扫描入口。

不允许：

- 主相机单独扫描一次
- 导星再扫描一次
- 极轴镜再扫描一次

正确流程应为：

- 扫描一次全部 QHY 相机
- 建立统一的 QHY 相机池
- 在池上做角色分配

### 3.2 自动建议不等于最终绑定

系统可以根据规则生成“建议分配草案”，但不能把草案直接当成最终结果。

最终绑定必须满足以下之一：

- 单相机单角色场景下直接确认
- 多相机场景下由用户确认后一次性应用

### 3.3 手工绑定优先级最高

只要用户手工选择过某个角色对应的相机，该角色即进入“手工锁定”状态。

在手工锁定状态下：

- 自动建议不得覆盖
- 重新连接时不得偷偷换机
- 导星、主拍摄、极轴业务链路不得再自行挑选其它相机

### 3.4 业务模块只读最终映射

一旦角色分配完成，后续所有业务模块都只能读取“最终角色映射”。

例如：

- 主拍摄只读 `MainCamera -> cameraId/handle`
- 导星只读 `Guider -> cameraId/handle`
- 极轴只读 `PoleCamera -> cameraId/handle`

后续业务模块不允许再次参与“选哪台相机”的决策。

## 4. 相机分类规则

### 4.1 所有 QHY 相机都是标准相机

所有 QHY 相机都进入同一个候选池。

包括但不限于：

- 真实 QHY 主摄
- 真实 QHY 导星机
- `DEMO` 相机

`DEMO` 不再作为“特殊设备类型”处理。

### 4.2 倾向分类

自动建议阶段，将 QHY 相机按“倾向”分为三类：

1. `主相机倾向`
2. `导星相机倾向`
3. `无倾向相机`

### 4.3 主相机倾向

典型特征：

- 分辨率较高
- 非导星类 5III 小相机
- 深空主摄类型号

用途：

- 自动建议时优先分配给 `MainCamera`

### 4.4 导星相机倾向

典型特征：

- `5III` 系列
- 分辨率较低
- 典型导星相机型号

用途：

- 自动建议时优先分配给 `Guider`

### 4.5 无倾向相机

典型包括：

- `DEMO`
- 无法明确识别用途的普通相机

用途：

- 不优先偏向主相机或导星
- 作为通用补位候选
- 可以被分配给任意普通相机角色

### 4.6 DEMO 的归类

`DEMO` 相机属于：

- 标准相机
- 无倾向相机

其含义是：

- 可以作为 `MainCamera`
- 可以作为 `Guider`
- 可以作为其它普通相机角色
- 自动建议时不享有主摄或导星倾向优先级

## 5. PoleMaster 规则

`PoleMaster` 不纳入普通相机分配流程。

### 5.1 唯一保留规则

只要扫描到 `PoleMaster` 类设备，且当前启用了 `PoleCamera` 角色，则：

- 直接自动绑定到 `PoleCamera`
- 不进入人工候选面板
- 不参与 `MainCamera` / `Guider` 的候选竞争

### 5.2 不做人工选择

`PoleMaster` 由于用途唯一，不建议出现在“普通相机人工选择面板”中。

这可以避免：

- 用户误把 PoleMaster 分给主相机
- 主相机或导星相机和 PoleMaster 抢设备

## 6. 角色范围

本文只讨论以下 3 个相机角色：

- `MainCamera`
- `Guider`
- `PoleCamera`

其中：

- `MainCamera` 和 `Guider` 参与“普通相机分配”
- `PoleCamera` 优先消耗 `PoleMaster`

## 7. 从全空状态开始的完整流程

### 7.1 初始状态

前提：

- 左侧设备栏全空
- 没有任何相机角色已连接
- 没有已生效的 QHY SDK 角色绑定

### 7.2 用户先选择厂商与连接模式

对于 QHY 相机，用户需要先在角色级别选择：

- 厂商：`QHYCCD`
- 连接方式：`INDI` 或 `SDK`

本文仅讨论：

- `QHYCCD + SDK`

### 7.3 收集角色需求

系统先只记录“哪些角色要走 `QHYCCD + SDK`”。

例如：

- `MainCamera = QHYCCD + SDK`
- `Guider = QHYCCD + SDK`
- `PoleCamera = 不启用`

此时不要立即做角色绑定。

### 7.4 统一扫描

当第一个 `QHYCCD + SDK` 角色出现时，系统创建统一会话并执行：

1. 扫描所有 QHY 相机
2. 获取全部 `cameraId`
3. 逐台打开并读取基础信息
4. 建立统一的相机池

输出应是：

- `cameraPool`

而不是“某个角色已经抢占了一台相机”。

## 8. 数据模型建议

### 8.1 QhyCameraInfo

建议字段：

- `cameraId`
- `displayName`
- `model`
- `serial`
- `isDemo`
- `isPoleMaster`
- `biasType`
- `maxWidth`
- `maxHeight`
- `pixelCount`
- `isOpened`
- `handle`
- `isAssigned`
- `assignedRole`

其中：

- `biasType` 取值建议为：
  - `MainPreferred`
  - `GuiderPreferred`
  - `Neutral`

### 8.2 QhyRoleRequest

建议字段：

- `role`
- `enabled`
- `manufacturer`
- `connectionMode`
- `manualLocked`
- `lockedCameraId`

### 8.3 QhyAllocationDraft

表示自动建议或用户编辑中的草案。

建议字段：

- `MainCamera -> cameraId/null`
- `Guider -> cameraId/null`
- `PoleCamera -> cameraId/null`
- `source`

其中 `source` 可为：

- `auto`
- `manual`
- `mixed`

### 8.4 QhyAllocationFinal

表示正式生效的绑定结果。

建议字段：

- `MainCamera -> cameraId/handle`
- `Guider -> cameraId/handle`
- `PoleCamera -> cameraId/handle`
- `lockState`

后续所有拍摄业务都只能读这里。

## 9. 自动建议规则

### 9.1 总体顺序

自动建议建议按以下顺序执行：

1. 先处理 `PoleMaster`
2. 再对剩余相机做普通角色分配

### 9.2 PoleMaster 分配

如果：

- 扫描到 `PoleMaster`
- 并且 `PoleCamera` 启用

则：

- `PoleCamera` 直接绑定 `PoleMaster`
- 将该设备从普通候选池中移除

### 9.3 MainCamera 分配

优先级建议：

1. 用户手工锁定
2. 主相机倾向相机
3. 无倾向相机
4. 导星倾向相机

### 9.4 Guider 分配

优先级建议：

1. 用户手工锁定
2. 导星倾向相机
3. 无倾向相机
4. 主相机倾向相机

### 9.5 Neutral 补位规则

当目标倾向设备不足时：

- 无倾向相机用于补位

这也是 `DEMO` 发挥作用的主要方式。

### 9.6 DEMO 不做特殊降权规则

由于 `DEMO` 已被定义为“无倾向相机”，它本身不再需要额外的特殊规则。

只需遵守：

- 不属于主摄倾向
- 不属于导星倾向
- 在对应倾向设备不足时自然补位

## 10. 按相机数量分情况讨论

### 10.1 扫描到 0 台相机

结果：

- 不生成最终绑定
- 不发送连接成功
- 前端提示“未发现 QHY 相机”

### 10.2 扫描到 1 台相机

#### 10.2.1 只启用 1 个角色

例如只启用 `Guider`：

- 直接建议该相机分配给 `Guider`
- 可直接确认后应用

#### 10.2.2 启用多个角色

例如启用 `MainCamera + Guider`：

- 默认只给优先角色分配
- 其余角色保持 `unassigned`
- 不允许同一 SDK 相机同时充当多个相机角色

### 10.3 扫描到 2 台相机

#### 10.3.1 只启用 1 个角色

- 生成建议草案
- 等待用户确认

#### 10.3.2 启用 2 个角色

例如：

- `MainCamera + Guider`

自动建议：

- `MainCamera` 优先主相机倾向
- `Guider` 优先导星倾向
- 倾向不足时由无倾向补位

### 10.4 扫描到 3 台及以上相机

流程同样是：

1. 自动生成草案
2. 用户确认或修改
3. 一次性应用

## 12. 分配面板建议

### 12.1 面板定位

分配面板不应再是当前这种“左侧角色列表 + 右侧候选列表 + 交换逻辑”的复杂弹窗。

建议改为：

- 左侧设备选择栏内嵌的小型候选条
- 面向单角色的轻量选择器
- 用户只需在候选相机中点选其一

### 12.2 面板职责

当某个角色存在多个候选相机时：

- 不弹独立大面板
- 在左侧设备选择栏中、对应角色的同等高度区域，出现一个小面板
- 小面板按横向排列方式展示多个候选相机
- 用户点击其中一个候选相机后，即完成该角色选择

这个小面板的职责不是“设备分配总控台”，而只是：

- 为当前角色展示候选相机
- 让用户从多个候选中选一个

### 12.3 候选相机展示方式

每个候选相机建议做成一个等宽小卡片，横向排列。

单个候选卡片内只保留两行文字：

1. 第一行：相机型号
2. 第二行：GUID 前缀信息

其中第二行可展示为：

- GUID 前几个字符
- 或 cameraId 前缀
- 目的是在多台同型号相机并存时提供区分度

不建议再显示：

- 冗长的完整设备名
- 设备类型标签
- “已绑定到谁”
- “可交换”
- “交换目标”

### 12.4 PoleMaster 的界面例外

`PoleMaster` 不进入这个候选条 UI。

因为：

- 它的归属唯一
- 只要存在就自动分配给 `PoleCamera`
- 不需要用户选择

因此左侧设备栏对 `PoleCamera` 的表现应为：

- 自动显示已选中的 PoleMaster
- 不出现候选横向选择条

### 12.5 触发条件

只有在“当前角色有多个普通候选相机”时，才显示该横向候选条。

分情况：

- `0` 个候选：显示未发现设备
- `1` 个候选：直接自动选中，不显示候选条
- `2+` 个候选：显示横向候选条，等待用户点击

### 12.6 取消交换逻辑

新 UI 不需要“交换”逻辑。

明确要求：

- 不显示“swappable”
- 不显示“bound to”
- 不显示“swap with”
- 不允许用户通过一次点击触发角色互换

如果某角色需要改选：

- 直接重新选择该角色的候选相机
- 后端按“该角色重新绑定”处理
- 不再使用“MainCamera 和 Guider 互换”这类隐式逻辑

### 12.7 应用时机

建议流程：

1. 后端确定“当前正在选择的角色”
2. 后端发送该角色的候选相机列表
3. 前端在左侧对应位置展示横向候选条
4. 用户点击某一个候选卡片
5. 前端发送“该角色选择了哪个 cameraId / poolIndex”
6. 后端应用该角色的新绑定

这里保留“按角色逐个确认”的轻量交互，不要求用户面对一个复杂总面板。

### 12.8 禁止边点边做角色互换

虽然 UI 上是“点一个候选就完成选择”，但后端仍需遵守：

- 不再自动做角色 swap
- 不再因为目标相机当前属于别的角色，就隐式交换两边角色
- 只处理“当前角色绑定到所选相机”
- 若目标相机原先属于其它角色，应先解除旧归属，再显式重建新归属

换句话说：

- 允许“重选”
- 不允许“交换”

## 13. 锁定规则

角色绑定建议分为三种状态：

1. `Unassigned`
2. `AutoAssigned`
3. `ManualLocked`

其中：

- `ManualLocked` 不允许被自动规则覆盖
- `AutoAssigned` 可以在重新配置时重新建议
- `Unassigned` 继续参与分配

## 14. 后续业务链路要求

### 14.1 主拍摄

只读：

- `Final.MainCamera.cameraId`
- `Final.MainCamera.handle`

### 14.2 导星

只读：

- `Final.Guider.cameraId`
- `Final.Guider.handle`

不得再按“5III 倾向”重新挑机。

### 14.3 极轴

只读：

- `Final.PoleCamera.cameraId`
- `Final.PoleCamera.handle`

## 15. 当前代码存在的主要偏差

与本文设计相比，现状主要偏差包括：

1. 主相机、导星、极轴镜存在多套独立 SDK 扫描与回绑逻辑
2. 自动建议和最终绑定混在一起
3. 多相机场景下即使已经自动绑定，仍继续弹出分配窗
4. `BindingDevice` 默认会触发角色交换
5. 导星链路可能再次按导星倾向改写最终设备
6. `DeviceIndiName` 同时承担 INDI 设备名与 SDK `cameraId` 语义

## 16. 推荐实施顺序

### 第一步

先建立统一的 `QHY camera pool` 与 `allocation draft` 数据结构。

### 第二步

将 `PoleMaster` 自动保留逻辑独立出来。

### 第三步

引入“主相机倾向 / 导星相机倾向 / 无倾向相机”分类。

### 第四步

把主相机、导星、极轴镜的 SDK 自动分配逻辑收敛到单一入口。

### 第五步

将前端分配面板改成“草案确认 -> 一次性应用”模式。

### 第六步

让主拍摄、导星、极轴业务全部改为只读最终映射。

## 17. 结论

`DEMO` 相机不应再被当作特殊测试设备，而应被视为“标准相机中的无倾向相机”。

`PoleMaster` 则应作为唯一性设备直接保留给 `PoleCamera`，不参与人工选择。

在此基础上，QHY SDK 多相机的正确方向应为：

- 统一扫描
- 统一建立候选池
- 基于倾向生成草案
- 用户确认后一次性应用
- 后续所有业务只读最终角色映射

后续代码实施建议严格以本文为准，避免再次回到“多个角色各自扫描、各自改绑、各自自动挑机”的混乱状态。

## 20. 实施进展（2026-07-07，后端第二版相机分配语义收口）

本轮后端继续收口了相机场景的分配语义，重点是把“自动绑完继续弹窗”和“隐式 swap”这两类混乱行为先压掉。

### 已完成内容

1. SDK 相机重新分配改为“抢占后清空旧角色”

位置：

- `QUARCS_QT-SeverProgram/src/mainwindow_device_connection.cpp`

当前行为：

- 若用户把某台 SDK 相机重新分配给 `MainCamera / Guider / PoleCamera` 中的另一个角色
- 后端不再做“两个角色互换”
- 改为：
  - 当前角色获得该相机
  - 原占用角色被清空为未分配

这与新的前端候选条交互是一致的。

2. INDI 相机场景也移除了隐式 swap

当前 INDI 的 `MainCamera / Guider / PoleCamera` 角色之间也不再做自动交换。

改为：

- 目标角色绑定新设备
- 原占用角色直接解绑

3. SDK 单独连接路径不再“自动绑上后继续弹窗”

当前以下路径已收口：

- `ConnectDriver(MainCamera, SDK)`
- `ConnectDriver(Guider, SDK)`
- `ConnectDriver(PoleCamera, SDK)`

行为改为：

- 若成功自动绑定，则直接成功
- 不再因为“还有多台候选相机”而继续发送 `ShowDeviceAllocationWindow`

4. SDK 待分配路径不再依赖旧弹窗

当前待分配时改为：

- 发送 `AddDeviceType`
- 发送 `ConnectDriverPendingAllocation:<role>`
- 保持候选相机列表消息

而不再对相机场景强制发送旧的独立分配窗口。

5. ConnectAllDeviceOnce 的相机场景也改为内嵌候选条思路

当前“连接全部设备”在遇到多相机且仍有角色未分配时：

- 发送相机角色的 `AddDeviceType`
- 对需要人工选择的角色发送 `ConnectDriverPendingAllocation:<role>`
- 发送候选相机列表

不再强制发起旧的弹窗流程。

本轮收口后，仍未完全完成的部分包括：
