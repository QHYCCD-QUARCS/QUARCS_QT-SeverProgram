# 2026-06-25 导星星点侦测调试阶段总结

## 1. 当前结论

本轮调试已经把“原图标注链路”和“前两层候选语义”基本理顺：

- 紫框：
  使用 `GuidingStarDetector` 平场分支里的
  `top2000 brightest corrected pixels + 5x5 dedup`
  语义，当前这层已经恢复正常。
- 黄框：
  保留紫框候选生成方式不变，只把 `SNR` 判定改成更接近单独试验程序的全局背景/噪声估计。
  当前黄框已经可以正常出现。
- 绿框：
  表示通过后续筛选后的最终候选。
- 青框：
  表示最终被选中的导星星点。

当前后端已经不是“只在第一次选星时画一次框”，而是会在后续帧里持续重算并重画调试框。

## 2. 已确认正确的部分

### 2.1 原图标注链路

现在调试框不是前端单独叠加，也不是从 `/dev/shm/guiding.fits` 回读另一帧再画。

当前实现是：

1. `GuiderCore` 在参与侦测的那一帧上完成候选计算
2. 通过 `requestPersistGuidingFitsAnnotated(...)` 把
   - 当前帧图像
   - 当前帧 dedup / snr / final 候选
   - 当前选中星
   一起发给 `MainWindow`
3. `MainWindow` 直接在这一帧原图上烧录紫/黄/绿/青框
4. 再生成导星预览 JPG 发给前端

因此“框和图像错一帧”的问题已经修复。

### 2.2 紫框语义恢复

后续调试中曾经把默认路径切到 `FlatFieldDetector`，导致紫框语义变成“全局局部极大值候选”，数量巨大且大量落在背景上。

现在已经恢复为默认走 `GuidingStarDetector` 路径，原因是这条路径更符合当前调试需求：

- 先 `3x3` 中值滤波
- 再 `boxFilter` 平场
- 再 `corrected = smoothed - flat + offset`
- 再取亮度前 `2000` 个像素
- 再做 `5x5` 临近去重

这也正是当前紫框的定义。

### 2.3 黄框层恢复

之前黄框完全没有，主要问题在于 `GuidingStarDetector` 里的旧 `SNR` 算法过于局部，近似是一个小窗口上的单点 z-score。

现在已经改为更接近单独试验程序的方式：

- 对扣平场结果先 `clip -> uint16`
- 在整幅扣平场图上估计全局背景 `bg`
- 在整幅扣平场图上估计全局噪声 `noise`
- `snr = (peak - bg) / noise`

这次改动后，真机日志中已经能看到黄框层恢复，例如：

- `dedupCandidates=224`
- `snrCandidates=224`
- `candidates=26`
- `selected=1`

并且已经可以稳定选出导星星点。

## 3. 当前实时刷新行为

之前的行为是：

- 只在 `Selecting` 阶段计算一次紫/黄/绿框
- 后续 `Looping / Calibrating / Guiding` 只是继续画缓存下来的那一批框

因此用户会看到“标注永远不动”。

现在已经改成：

- `Selecting` 阶段继续计算并画框
- `Looping / Calibrating / Guiding` 阶段也会每帧重新计算候选并重画

也就是说，当前看到的调试框会随着当前导星帧实时更新。

真机验证中，连续两次选中星位置已经出现变化，例如：

- 一次为 `x=3201.52 y=288.31`
- 后续更新为 `x=3238.88 y=357.98`

说明后续帧确实在重新计算并重画，而不是继续沿用第一次选星时的静态框。

## 4. 本轮关键实现选择

本轮最终没有采用“简单切换到 `FlatFieldDetector`”的方式，而是做了融合：

- 候选生成：
  继续使用 `GuidingStarDetector`
  的 `top2000 + dedup` 路径
- `SNR` 判定：
  借鉴并部分对齐单独试验程序的全局背景/噪声思路
- 调试可视化：
  统一由后端直接烧录到当前原图

这样做的原因是：

- 紫框需要保留“前 2000 + 去重”的诊断意义
- 黄框需要更可靠的 `SNR` 判定
- 同时避免再次引入“紫框语义改变”的混乱

## 5. 当前尚未完成的部分

当前真正剩下的重点问题已经从“为什么没有黄框”转移到：

`为什么黄框很多，但绿框只剩二十几个`

也就是：

- `dedupCandidates` 很多
- `snrCandidates` 也很多
- 但 `candidates` 明显减少

因此下一轮调试重点应转到黄框之后的后段筛选，包括：

- `HFD` 过滤
- 边缘过滤
- 质心筛选
- 重复候选剔除

## 6. 建议的下一轮调试入口

下一轮建议直接围绕“黄框到绿框”的损失链路做分解，优先看：

1. 每个候选是被哪一条后段条件淘汰的
2. `HFD` 分布是否异常
3. 重复候选剔除是否过强
4. 边缘过滤是否误伤
5. 当前最终 `selected` 是否总是来自同一类星形

如果继续做可视化，建议下一步优先考虑：

## 7. 本轮耗时优化收口

### 7.1 当前阶段结论

截至本轮结束，导星平场选星链路已经达到“基本可用”的速度水平。

当前在树莓派 4B 真机上的典型耗时已经从最初的 `8.3s+` 下降到约 `0.39s ~ 0.42s`。

这是一个阶段性可接受结果，因此本轮工作先以“收口并记录”为主，后续如需继续深挖，可再围绕剩余热点单独展开。

### 7.2 关键耗时演进

本轮过程中，导星选星耗时大致经历了下面几个阶段：

1. 初始版本
   - 单帧总耗时约 `8.3s ~ 8.6s`
   - 最大瓶颈是 `globalBgNoise`
   - `globalBgNoise` 约 `6.1s`
   - `pixelCollection` 约 `1.5s`
   - `peakSelection` 约 `0.27s`

2. 改为局部环形背景/局部方差后
   - 去掉整帧 `globalBgNoise`
   - 单帧总耗时下降到约 `2.0s ~ 2.3s`
   - 主要瓶颈转移为：
     - `pixelCollection`
     - `peakSelection`

3. 改为 `16bit` 分桶 + 单桶比例上限后
   - 单帧总耗时下降到约 `0.79s ~ 0.88s`
   - `peakSelection` 基本降为 `3ms`
   - `pixelCollection` 下降到约 `0.32s ~ 0.37s`

4. 树莓派原生构建切到 `Release` 后
   - 单帧总耗时下降到约 `0.47s ~ 0.51s`
   - 实际编译参数确认为：
     - `CMAKE_BUILD_TYPE=Release`
     - `-O3 -DNDEBUG`

5. 去掉 `clone + max + min`，直接 `convertTo(CV_16U)` 后
   - 单帧总耗时进一步下降到约 `0.39s ~ 0.42s`
   - `clip+convertTo16u` 从原来的 `107ms ~ 128ms`
     下降到约 `24ms ~ 26ms`

### 7.3 当前最新典型耗时

当前最新版本的导星平场选星链路，真机日志中的典型阶段耗时如下：

- `medianBlur+convertTo32f`：约 `70ms ~ 87ms`
- `boxFilter`：约 `134ms ~ 144ms`
- `subtractFlat`：约 `49ms ~ 56ms`
- `clip+convertTo16u`：约 `24ms ~ 26ms`
- `pixelCollection`：约 `69ms ~ 74ms`
- `peakSelection`：约 `0ms`
- `dedup+snr+hfd`：约 `3ms`

当前总耗时主流落在：

- `387ms`
- `397ms`
- `399ms`
- `417ms`
- `418ms`

### 7.4 本轮已完成的关键优化

#### 7.4.1 去掉整帧全局背景噪声统计

之前 `globalBgNoise` 的实现方式是：

- 把整幅扣平场图所有像素塞入 `vector`
- 排序
- 估计全局背景和噪声

该路径在树莓派上耗时超过 `6s`，是最初的绝对瓶颈。

后来改为：

- 每个候选点使用局部环形背景
- 每个候选点使用局部标准差

这一步直接砍掉了最大的性能问题，也更贴近最早独立试验脚本的思路。

#### 7.4.2 候选提取从“全量收集 + partial_sort”改为“16bit 分桶”

之前 `pixelCollection + peakSelection` 的逻辑是：

- 收集所有大于 `offset` 的像素
- 形成大 `vector`
- 再做 `partial_sort` 取前 `2000`

这条路径的主要问题是：

- 实际会收集数百万个像素
- 大量内存写入
- 排序成本高

后续改为：

- 对 `uint16` 亮度值做分桶
- 每个桶设置上限
- 单桶上限与总候选预算联动
- 再从高亮桶向下累计拿到前 `2000` 个候选

当前实现参数为：

- 总候选预算：`2000`
- 单桶比例上限：`50%`
- 即单桶最多保留：`1000`

这一步使得：

- `peakSelection` 基本降到 `0ms ~ 3ms`
- `pixelCollection` 明显下降
- 同时降低了大面积饱和区域垄断候选的风险

#### 7.4.3 切换树莓派原生构建为 `Release`

本轮确认了此前树莓派原生部署脚本并未显式指定 `Release`，实际构建一度是：

- `CMAKE_BUILD_TYPE` 为空
- 实际 `CXX_FLAGS` 没有 `-O2/-O3`

后续已修改部署脚本，默认使用：

- `BUILD_TYPE=Release`
- `cmake -DCMAKE_BUILD_TYPE=Release ..`

并已在真机确认：

- `CXX_FLAGS = -O3 -DNDEBUG -fPIC -std=gnu++17`

#### 7.4.4 简化 `clip -> uint16` 转换链路

之前代码为：

- `clone`
- `cv::max(..., 0.0, ...)`
- `cv::min(..., 65535.0, ...)`
- `convertTo(CV_16U)`

后续验证当前 OpenCV 路径下，直接：

- `corrected.convertTo(corrected16, CV_16U)`

即可满足所需饱和裁剪行为。

这一步将 `clip+convertTo16u` 从 `100ms+` 降到了 `25ms` 左右。

### 7.5 当前剩余热点

截至本轮结束，当前真正剩下的热点已经比较集中：

1. `boxFilter`
2. `medianBlur+convertTo32f`
3. `pixelCollection`

其中：

- `boxFilter` 目前看更像是真实滤波成本
- `medianBlur+convertTo32f` 更像必要前处理成本
- `pixelCollection` 仍有一些桶容器和写入成本，但已经不再是压倒性问题

### 7.6 关于 GPU / OpenCL 加速的阶段性评估

本轮也对“是否可以依赖其他处理器资源继续加速”做了一个阶段性评估。

当前树莓派 4B 环境实测情况：

- CPU：`4x Cortex-A72`
- OpenCV：`3.4.14 Release`
- 已启用：
  - `NEON FP16`
  - `carotene`
  - `pthreads`
- OpenCL 运行态实测为：
  - `haveOpenCL = False`
  - `useOpenCL = False`

因此当前结论是：

- 现阶段真正可用的主要仍是 ARM CPU 的 SIMD / OpenCV CPU 路径
- GPU / OpenCL 当前并没有形成可直接落地的执行链路
- 如果未来要继续做更重的硬件加速，需要先单独评估树莓派上的 OpenCL / Vulkan / 其他 GPGPU 条件

换句话说，当前最现实的加速基础仍然是：

- `Release`
- `O3`
- ARM `NEON`
- OpenCV 已有的 CPU 优化实现

### 7.7 本轮建议收口点

本轮工作建议到此先阶段性结束，原因如下：

1. 当前选星耗时已经进入 `0.4s` 量级，具备实际使用价值
2. 本轮已完成多轮高收益优化，收益曲线开始变缓
3. 当前剩余瓶颈更多是滤波类真实计算成本，不再是明显的低垂果实

如果后续继续推进，建议的下一轮方向是：

1. 继续评估 `boxFilter` 是否存在更合适的 CPU 侧实现路径
2. 评估 `medianBlur + boxFilter` 是否值得做更底层的 ARM/NEON 定制优化
3. 若要进一步压榨极限，再单独研究树莓派上是否存在可靠的 GPU / OpenCL / 其他协处理路径
