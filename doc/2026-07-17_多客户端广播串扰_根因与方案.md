# 多客户端广播串扰：现象、根因与解决方案

> 2026-07-17。起因：重构设备连接代码期间，测试脚本与浏览器同时连着，候选相机
> 从 4 台被放大到 48 台。查下来不是相机扫描的问题，是**消息架构**的问题。
> 这个问题在"多人同时打开前端"时必然发作，不是测试环境特有的。

## 1. 现象与实测证据

| 现象 | 实测 |
|---|---|
| 候选列表被撑大 | 测试脚本 + 浏览器同开时，脚本观察到的候选从 4 条涨到 **48** 条 |
| 别人操作弹出我的面板 | A 点"连接"，B 的前端也弹出相机分配条 |
| 前端自动逻辑被别人触发 | B 收到 A 的回执后，B 的前端也发命令 → 命令风暴 |

转发器日志能直接看出有几个客户端在发命令（`msgid` 是天然的客户端指纹：
前端用 `<时间戳>-<序号>`，测试脚本用自定的前缀）。实测确认当时有 3 个客户端在发命令。

> 关于那个 48：这是**测试脚本**看到的条数。前端侧另有一层按 `CCD:<index>` 的
> upsert 去重（`f9ddc6b31`），所以纯粹的消息重复并不会把前端列表撑大——48 意味着
> **不同的 index 真的变多了**（多客户端各自触发连接 → INDI 设备被反复加入）。
> 两件事要分开看：消息重复投递（下面 2.2/2.3）与设备表增长是不同的病灶。

## 2. 根因

三个原因叠加，缺一不可，因此**只修一个都不彻底**。

### 2.1 转发器是"哑总线"，不是路由器

`QUARCS_NodeJs-Transponder/server.js`：

```js
// 默认逻辑：迭代所有客户端并广播 WebSocket 消息
wss.clients.forEach(function each(client) {
  if (client.readyState === WebSocket.OPEN && client.id !== ws.id) {
    client.send(data, { binary: isBinary });   // 发给"除发送者外的所有人"
  }
});
```

它**不区分谁是后端、谁是前端**——任何人说的话，转给其他所有人。于是两个方向都漏：

- 前端 A 的 `Vue_Command` → 后端收到，**前端 B 也收到**；
- 后端对 A 的回执 → A 收到，**B 也收到**。

### 2.2 一条消息同时承担两种语义

后端下发的消息其实是两类，但在协议里长得一模一样：

| 语义 | 该发给谁 | 例子 |
|---|---|---|
| **状态通知**（事实变了） | 所有人（多人观看时状态本就该一致） | `BindDeviceTypeList:...` |
| **请求回执**（回答"你刚点的那下"） | **只发给发起者** | `DeviceToBeAllocated:...`、`ConnectDriverSuccess:...` |

`DeviceToBeAllocated` 是回执，却被广播给所有人 → B 弹出自己没点过的面板。
这与本次重构中反复出现的根因是同一个模式：**一个东西承载两种含义**
（`DeviceIndiName` = 当前绑定 vs 上次记忆；`ConnectDriverSuccess` = 驱动已连 vs 角色已绑）。

### 2.3 回执在协议层就无法归属

- 后端下发用 `WebSocketThread::sendMessageToClient(QString message)`——**裸字符串，不带 msgid**；
- 后端代码里 **grep 不到任何 msgid 的读取**：msgid 纯粹是前端自己的约定，转发器盲转、后端根本不看。

所以就算想定向，转发器也**没有任何依据**知道这条回执是回给谁的。

### 2.4 候选列表由"增量消息"拼装，只能加不能减

候选是一台相机一条 `DeviceToBeAllocated:CCD:<idx>:<name>:<cat>`（后端 **26 处**在发），
前端 upsert 进列表。列表的"当前全貌"不是后端说的，而是前端**攒**出来的。

于是"移除"没有对应的消息可表达。现在靠的是：**点连接时前端先整个清空**
（`connectDriver()` 里 `clearInlineCameraAllocationState('all')`），再等新的候选填回来。
这在单客户端下能凑效，但：

- **多客户端下直接破**：B 没点过连接 → B 从不清空 → 却持续收到 A 的候选 →
  B 的列表只增不减，越攒越偏离真相；
- 即使单客户端，"先清空、再等消息填回"本身是个时间窗（清空是发命令时，
  候选是若干秒后才到），中间任何一条别人的消息都会混进来。

> 根因不是"重复投递"，而是**列表的权威来源被放在了前端**——
> 后端只会说"再加一台"，从没有任何一条消息能说"当前就这些"。

## 3. 为什么现在就要修

设备是**单台望远镜的共享独占资源**。两个用户同时点 MainCamera 的"连接"，
本身就是冲突操作。当前架构下这个冲突**没有任何一层去仲裁**，而是让所有客户端
一起收到彼此的回执，各自的自动逻辑再互相触发——这正是我们这次重构反复在拆的
"反馈环"的源头。设备连接代码刚被理顺，若不修这一层，同样的紊乱会从消息层再长回来。

## 4. 解决方案（按性价比排序）

### 方案 A：候选列表改为全量快照（建议先做）

把"加一条候选"改成"这是当前完整的候选列表"：

```
CandidateList:CCD:[{idx,name,cat}, ...]      # 一条消息 = 全量，替换而非追加
```

- **权威来源回到后端**：前端不再"攒"列表，收到什么就显示什么 → 2.4 直接消失；
- **天然幂等**：重复投递、多客户端、重复点击，结果都一样；
- 顺带干掉两处补丁：前端的 upsert 去重、以及"点连接先清空"的时间窗；
- 后端 26 处发射点收敛成 1 处 `reportCandidates()`——和本次重构一直在做的
  "同一个概念只在一个地方实现"是同一件事（扫描收敛到 `registerSdkCameraPool`、
  绑定收敛到 `bindDeviceToRole` 都是这个套路）；
- 不依赖 B，先做不亏。

### 方案 B：回执定向（msgid 回传 + 转发器单播）

区分两种语义，让协议自己表达：

```jsonc
{"type":"event",  "message":"BindDeviceTypeList:..."}                 // 广播
{"type":"reply",  "msgid":"<原请求id>", "message":"ConnectDriverSuccess:..."}  // 只回发起者
```

改动三处：
1. **后端**：处理命令时记住当前 msgid，回执带上它（`sendMessageToClient` 加一个
   msgid 参数，或用一个"当前请求上下文"）；
2. **转发器**：维护 `msgid → 发起者 socket`，`type:"reply"` 单播，`type:"event"` 广播；
3. **前端**：忽略 msgid 不是自己的 reply（双保险）。

比 A 贵，但它治的是 2.2/2.3——**回执与通知的语义分离**，是真正的架构修复。

### 方案 C：控制权独占（长期，真多用户才需要）

一个客户端持"控制权"，其余只读观看；控制权可移交。
只有明确要支持"多人同时操作一台望远镜"时才值得做。
**在做 A/B 之前不建议动 C**——C 解决的是"谁能操作"，A/B 解决的是"消息发给谁"，
是两个问题，C 不能替代 A/B。

## 5. 建议

**先 A，再 B，C 待定。**
A 用最小成本消掉最刺眼的叠加现象，且无论将来是否做 B 都不会白做；
B 是架构上的正解，但要动三个仓库，适合作为独立一轮；
C 是产品决策（要不要支持多人同时操作），不是技术债。

## 6. 对测试的影响（当前必须遵守）

在 A/B 落地之前，**跑 WS 集成测试时不能开着前端页面**——前端是另一个客户端，
会收到测试脚本的回执并做出反应，污染数据（实测把候选从 4 放大到 48）。
浏览器要**整页关闭**，只刷新无效（前端会自动重连）。
见 `src/tests/ws/README.md`。
